# 页面冲突 — TouchGFX Designer 与用户代码冲突问题全记录

## 核心矛盾

TouchGFX Designer 每次 "Generate Code" 会覆盖 `generated/gui_generated/` 下的所有文件，但不会动 `gui/` 下的用户代码。问题在于：

- **生成基类** (`FrontendApplicationBase`) 的方法集随 Designer 交互配置变化而增减
- **用户派生类** (`FrontendApplication`) 如果声明了同名方法，会导致类型冲突或找不到方法
- Designer 生成的代码有两类 Bug：重复添加 widget、回调类型硬编码为派生类

---

## 问题 1：SensorPage 通配符文本框在 Designer 重生成后丢失

**现象**：Designer 生成的是普通 `TextArea`，不支持 `setWildcard()` 动态更新传感器数据，编译报 20 个 `textArea1Buffer` 未定义。

**根因**：Designer 的 TextArea 不支持通配符模式，只生成 `TextArea` 类型。

**解决**：在用户代码中声明独立的 `TextAreaWithOneWildcard` 控件，不依赖生成基类。

**改动文件**：[`SensorPageView.hpp`](FS-STM32U575-Total(Release-v2)/TouchGFX/gui/include/gui/sensorpage_screen/SensorPageView.hpp)

```cpp
// 新增：通配符文本控件（替代生成基类的 textArea1-4）
touchgfx::TextAreaWithOneWildcard textModal1; // 温度
touchgfx::TextAreaWithOneWildcard textModal2; // 湿度
touchgfx::TextAreaWithOneWildcard textModal3; // CO2
touchgfx::TextAreaWithOneWildcard textModal4; // 心率/血氧

static const uint16_t TEXTAREA1_SIZE = 30;
touchgfx::Unicode::UnicodeChar textArea1Buffer[TEXTAREA1_SIZE];
static const uint16_t TEXTAREA2_SIZE = 30;
touchgfx::Unicode::UnicodeChar textArea2Buffer[TEXTAREA2_SIZE];
static const uint16_t TEXTAREA3_SIZE = 30;
touchgfx::Unicode::UnicodeChar textArea3Buffer[TEXTAREA3_SIZE];
static const uint16_t TEXTAREA4_SIZE = 40;
touchgfx::Unicode::UnicodeChar textArea4Buffer[TEXTAREA4_SIZE];
```

**永久性**：Designer 永远不会修改 `gui/` 目录，通配符控件不会丢失。

## 问题 2：HomePage 扩展按钮（toggleButton4）跳转丢失

**现象**：Designer 重生成后，toggleButton4 按钮跳转到 ApplicationPage 而不是 SensorPage。

**根因**：生成基类 `HomePageViewBase` 中的 `buttonCallback` 被 Designer 重置。

**解决**：在构造函数和 `setupScreen()` 都覆盖按钮动作。两处都写是为了防御 Designer 在不同时机重置。

**改动文件**：[`HomePageView.cpp`](FS-STM32U575-Total(Release-v2)/TouchGFX/gui/src/homepage_screen/HomePageView.cpp)

```cpp
HomePageView::HomePageView()
    : expandButtonCallback(this, &HomePageView::expandButtonCallbackHandler)
{
    toggleButton4.setAction(expandButtonCallback); // 第一次覆盖
}

void HomePageView::setupScreen()
{
    HomePageViewBase::setupScreen();
    toggleButton4.setAction(expandButtonCallback); // 第二次覆盖，防 Designer 重置
}

void HomePageView::expandButtonCallbackHandler(const touchgfx::AbstractButton& src)
{
    application().gotoSensorPageScreenNoTransition(); // 跳转到 SensorPage
}
```

**永久性**：用户代码，Designer 不碰。

## 问题 4：点击图标进入 modal 容器后，图标没有全部隐藏

**现象**：打开一个模态窗口，其他传感器图标仍然显示。

**根因**：`showTemp()` / `showHum()` / `showCO2()` / `showHeartRate()` 中只显示 modal，未隐藏图标。

**解决**：新增 `setIconsForModal(int modal)` 方法，`modal >= 0` 时全部隐藏，`modal < 0` 时全部显示。

**改动文件**：[`SensorPageView.cpp`](FS-STM32U575-Total(Release-v2)/TouchGFX/gui/src/sensorpage_screen/SensorPageView.cpp)

```cpp
void SensorPageView::setIconsForModal(int modal)
{
    bool showAll = (modal < 0);
    scalableImage1.setVisible(showAll); // 温度图标
    scalableImage2.setVisible(showAll); // 湿度图标
    scalableImage3.setVisible(showAll); // CO2 图标
    scalableImage4.setVisible(showAll); // 心率图标
}
```

每个 `show*()` 方法调 `setIconsForModal(n)`，`hideActiveModal()` 调 `setIconsForModal(-1)`。

## 问题 5：心率初始显示 51 bpm（传感器未接触皮肤）

**现象**：不碰 MAX30102 传感器，点开心率页面显示 51 bpm。

**根因**：MAX30102 算法 (`algorithm.c`) 在环境光噪声中检测到 ≥2 个伪峰，计算了假心率值。

**解决**：`showHeartRate()` 先将 buffer 重置为 `"--"`，等有效数据更新。`updateSensorInfo` 使用标记值 `0xFFFFFFFF` 作为无效哨兵，15 秒无有效数据保持死区。

**改动文件**：[`SensorPageView.cpp`](FS-STM32U575-Total(Release-v2)/TouchGFX/gui/src/sensorpage_screen/SensorPageView.cpp)

```cpp
void SensorPageView::showHeartRate()
{
    if (activeModal == 3) {
        modalWindow4.hide();
        activeModal = -1;
        setIconsForModal(-1);
    } else {
        if (activeModal >= 0) hideActiveModal();
        Unicode::strncpy(textArea4Buffer, "HR: -- bpm  SpO2: --", TEXTAREA4_SIZE);
        modalWindow4.show();
        activeModal = 3;
        setIconsForModal(3);
        textModal4.invalidate();
    }
}
```

## 问题 6：生成基类回调类型冲突（`&FrontendApplication::` vs `&FrontendApplicationBase::`）

**现象**：29 个编译错误，每个包含 `FrontendApplication.hpp` 的翻译单元都报错。

**根因**：Designer 在生成基类 `.cpp` 中硬编码了 `Callback<FrontendApplicationBase>(this, &FrontendApplication::method)`。当用户派生类声明了同名方法时，`&FrontendApplication::method` 解析为 `void (FrontendApplication::*)()`，与 `Callback<FrontendApplicationBase>` 要求的 `void (FrontendApplicationBase::*)()` 类型不兼容。

**C++ 原理**：
```
// 基类 Callback 要求 void (FrontendApplicationBase::*)()
// 派生类方法实际类型是 void (FrontendApplication::*)()
// C++ 不允许这种隐式转换（协变参数在函数指针中是反变的）
```

**解决**：`fix_after_designer.py` Fix 2，将生成基类中所有 `&FrontendApplication::` 替换为 `&FrontendApplicationBase::`。使用基类限定名后，函数指针类型始终正确。

**被修文件**（自动）：`generated/gui_generated/src/common/FrontendApplicationBase.cpp`

```python
old = "touchgfx::Callback<FrontendApplicationBase>(this, &FrontendApplication::"
new = "touchgfx::Callback<FrontendApplicationBase>(this, &FrontendApplicationBase::"
content = content.replace(old, new)
```

## 问题 7：`changeToStartScreen()` 调用的方法被 Designer 删除

**现象**：`FrontendApplication.hpp` 中 `changeToStartScreen()` 调用了 `gotostartuppageScreenNoTransition()`，但 Designer 把该方法替换成了 `gotoSensorPageScreenNoTransition()`，报 `use of undeclared identifier`。

**根因**：Designer 的起始页面设置决定了基类中 `changeToStartScreen()` 调哪个方法：
- 起始页 = startuppage → 基类有 `gotostartuppageScreenNoTransition()`
- 起始页 = SensorPage → 基类有 `gotoSensorPageScreenNoTransition()`，删掉 `gotostartuppage`
- 用户派生类手写了 `changeToStartScreen()` 覆盖，调用了已被删除的方法名

**解决**：不在派生类覆盖 `changeToStartScreen()`，接受 Designer 的起始页设置。需要自定义启动页时在 Designer 中配置。

**改动文件**：[`FrontendApplication.hpp`](FS-STM32U575-Total(Release-v2)/TouchGFX/gui/include/gui/common/FrontendApplication.hpp)

```cpp
class FrontendApplication : public FrontendApplicationBase
{
public:
    FrontendApplication(Model& m, FrontendHeap& heap);
    virtual ~FrontendApplication() { }

    virtual void handleTickEvent()
    {
        model.tick();
        FrontendApplicationBase::handleTickEvent();
    }

    // SensorPage 过渡 — 始终在用户代码中，因为 Designer 可能不生成
    void gotoSensorPageScreenNoTransition();
    void gotoSensorPageScreenNoTransitionImpl();

private:
    touchgfx::Callback<FrontendApplication> sensorPageTransitionCallback;
};
```

## 问题 8：Designer 重生成后 `gotoSensorPageScreenNoTransition` 时有时无

**现象**：同上一条，但针对 SensorPage。Designer 有时生成 `gotoSensorPageScreenNoTransition` 有时不生成，取决于交互配置。

**根因**：Designer 只为有交互的页面生成过渡方法。如果 SensorPage 在 Designer 中未设交互，就不生成。

**解决**：用户派生类始终声明并实现 `gotoSensorPageScreenNoTransition()`，不依赖生成基类。如果 Designer 也生成了，两个版本通过名字隐藏共存（C++ 合法），`fix_after_designer.py` Fix 2 保证基类回调使用 `&FrontendApplicationBase::` 不会有类型冲突。

**改动文件**：[`FrontendApplication.cpp`](FS-STM32U575-Total(Release-v2)/TouchGFX/gui/src/common/FrontendApplication.cpp)

```cpp
void FrontendApplication::gotoSensorPageScreenNoTransition()
{
    pendingScreenTransitionCallback = &sensorPageTransitionCallback;
}

void FrontendApplication::gotoSensorPageScreenNoTransitionImpl()
{
    touchgfx::makeTransition<SensorPageView, SensorPagePresenter, touchgfx::NoTransition, Model>(
        &currentScreen, &currentPresenter, frontendHeap, &currentTransition, &model);
}
```

## 自动化脚本：`fix_after_designer.py`

**位置**：[`TouchGFX/fix_after_designer.py`](FS-STM32U575-Total(Release-v2)/TouchGFX/fix_after_designer.py)

**用法**：Designer "Generate Code" → `python fix_after_designer.py`

**功能**：
| Fix | 说明 |
|-----|------|
| Fix 1 | 删除 `SensorPageViewBase.cpp` 中重复的 `add(__background)` |
| Fix 2 | 将 `FrontendApplicationBase.cpp` 中 `&FrontendApplication::` 改为 `&FrontendApplicationBase::` |

**防御**：自动解除 Designer 设置的只读属性（`chmod`）。

## 文件权限地图

| 目录 | Designer 行为 | 策略 |
|------|--------------|------|
| `gui/` | **永远不碰** | 放所有用户逻辑（通配符、按钮绑定、过渡方法） |
| `generated/gui_generated/` | **每次覆盖** | 放基类代码，Bug 靠脚本修 |
| `fix_after_designer.py` | 不碰 | 每次重生成后手动执行 |

---

## 设计原则

> **派生类不声明生成基类已有的方法。Designer 给什么用什么，只追加基类没有的。**
>
> **例外**：`gotoSensorPageScreenNoTransition` 必须由派生类提供，因为 Designer 是否生成它不可预测。

## 关键类型规则（C++ 备忘）

```
Callback<FrontendApplicationBase> 只能接受 void (FrontendApplicationBase::*)()
Callback<FrontendApplication>      只能接受 void (FrontendApplication::*)()

两者之间不兼容 — 即使有继承关系。
所以生成代码必须用 &FrontendApplicationBase:: 限定函数指针类型。
```
